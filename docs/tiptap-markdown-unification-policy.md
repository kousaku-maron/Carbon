# Carbon TipTap Markdown 統一方針（v3 最新化）

## 1. 目的

本方針は、Carbon の Markdown 変換経路を TipTap 系に統一し、以下を同時に満たすことを目的とする。

- 複数空行（連続改行）の保持をベストエフォートで改善する
- 保存/読込パイプラインのロジックをシンプルに保つ
- 既存の保存仕様（相対リンク、`carbon://asset/...`）を維持する

---

## 2. 背景（現行課題）

現行実装は以下の構成になっている。

- エディタ本体: TipTap
- Markdown 読込: `marked`（MD -> HTML）
- Markdown 保存: `turndown`（HTML -> MD）

現行構成の課題:

- 変換エンジンが分離しており、往復時の挙動差分を管理しづらい
- 空行や段落境界で正規化が起き、入力時の見た目と再読込時の結果がずれることがある
- 変換ロジックの責務が分散し、将来の拡張時に確認ポイントが多い

---

## 3. 対象範囲

### 3.1 対象

- TipTap v3 系（最新安定版）への移行
- Markdown I/O を TipTap 系（`@tiptap/markdown` + 必要な custom parser/serializer）へ統一
- 空行保持に関する回帰テスト整備（ベストエフォート基準）

### 3.2 非対象

- ノート切替時の未保存 flush 問題
- UI デザイン変更
- ファイル保存先や Vault の仕様変更

注記:
- 未保存 flush 問題は別トラックで扱う。本方針のスコープ外とする。

---

## 4. 方針

### 4.1 変換経路を 1 系統に統一する

- 読込: Markdown -> TipTap 文書
- 保存: TipTap 文書 -> Markdown
- `marked` / `turndown` の直接利用は段階的に廃止する

狙い:
- 変換責務を TipTap 側に集約し、デバッグと仕様管理を簡素化する

### 4.2 空行保持は「ベストエフォート」を明文化する

- 目標: 一般的な編集操作で「意図した空行」が過度に失われないこと
- 非目標: すべてのケースで「空行の個数」を厳密保証すること

理由:
- Rich Text 文書モデルを経由する以上、Markdown の生テキスト情報を完全可逆には扱えないケースがある

### 4.3 保存仕様の互換性を最優先する

- リンクは標準 Markdown の相対リンクを維持する
- 画像は `carbon://asset/...` を永続参照として維持する

具体策:
- parser/serializer をカスタムし、既存仕様を壊さない
- 既存ファイルを破壊的に再整形しない

### 4.4 依存更新は「最新版追従」ではなく「固定 + 検証」で運用する

- TipTap v3 系の最新安定版へ更新する
- ただし運用時はバージョン固定（lock）を行い、回帰確認後にのみ更新する

---

## 5. 実装ポリシー

### 5.1 Canonical Data Flow

以下を唯一の正規フローとする。

1. ファイル（Markdown）を読込
2. TipTap Markdown parser で文書化
3. TipTap で編集
4. TipTap Markdown serializer で Markdown 化
5. ファイルへ保存

### 5.2 カスタムルールの移植

現行で必要なルールは TipTap 側に移植する。

- task list 記法
- `carbon://asset/...` を利用する画像記法
- 必要に応じたリンク変換ルール

### 5.3 変換レイヤ分離

- 変換実装は 1 モジュールに集約する
- `NoteEditor` は「編集状態」と「保存トリガー」に責務を限定する

---

## 6. 段階移行（推奨）

### Phase 1: 準備

1. 空行・リスト・リンク・画像を含む回帰フィクスチャを追加
2. 期待結果（ベストエフォート基準）を明文化

### Phase 2: 基盤更新

1. TipTap v3 系へ更新
2. `@tiptap/markdown` を導入
3. 既存拡張（Link, TaskList, Image, Suggestion）の互換性を確認

### Phase 3: 統一

1. `marked/turndown` 依存の呼び出しを置換
2. custom parser/serializer へ task list / asset ルールを移植
3. 変換モジュール外からの直接変換呼び出しを禁止

### Phase 4: 収束

1. 不要依存（`marked`, `turndown`）を削除
2. ドキュメント更新
3. 手動回帰を完了して確定

---

## 7. 受け入れ基準

1. Markdown 変換経路が TipTap 系に一本化されている
2. 連続空行を含む主要ケースで、現行より改行崩れが減少している
3. 相対リンクと `carbon://asset/...` の保存仕様が維持されている
4. 変換ロジックが単一モジュールに集約されている
5. 既存ノート資産に対して破壊的変更が発生しない

---

## 8. リスクと対応

### 8.1 主なリスク

- v2 -> v3 で拡張 API の差分対応が必要
- parser/serializer カスタムの不足で互換性欠落が起こる
- 空行問題が一部ケースで残る可能性がある

### 8.2 対応方針

- フィクスチャベースの比較テストを先行する
- 主要ノートセットで手動確認を行う
- 仕様差分は「既知制約」として明示し、改善対象を継続管理する

## 10. 補足

本方針は、以下のプロダクト要件を維持したまま技術基盤を整理するためのものである。

- ローカル Markdown ファイル主権
- 標準 Markdown 互換を重視したリンク保存
- 非公開画像配信のための `carbon://asset/...` 永続参照
